# Beginner Code Reading By Question

## Question 1: What is the current mainline?

Read `docs/CURRENT_PROJECT_STATE.md`, `README.md`, `PROGRESS.md`, and
`docs/START_HERE.md`. The mainline is a closed repair loop: generate semantic
layout, observe state, review errors, plan repair, build mask, run semantic
repair when needed, and evaluate.

Current routing:

- Planner actions: ADD / REMOVE / TRANSLATE / ROTATE / SCALE / REPLACE
- semantic_repair4: ADD / REMOVE / TRANSLATE / REPLACE
- parametric_update: ROTATE / SCALE

## Question 2: What does the VLM Planner output?

Read current handoff docs, current schemas, and `tools/validate_current_repairplan.py`.
The VLM outputs one JSON RepairPlan with `action_type`, `execution_mode`,
target/source refs, `target_state_delta`, optional `parametric_delta`, optional
`mask_spec`, optional `correction_prompt`, `acceptance_criteria`, and confidence.

Action constraints:

- ADD: semantic repair, target region, target category/id/rgb, count +1.
- REMOVE: semantic repair, source region, source instance, count -1.
- TRANSLATE: semantic repair, old region plus new region, disconnected mask allowed.
- REPLACE: semantic repair, source region plus target region, source and target categories.
- ROTATE: structured update, target instance, rotation field, no mask or prompt.
- SCALE: structured update, target instance, size/scale field, no mask or prompt.

## Question 3: Why do ROTATE and SCALE not train Qwen/DiffSynth semantic repair?

The fixed-palette top-down semantic map does not explicitly encode hidden yaw or
structured size fields as separate semantic channels. ROTATE and SCALE are legal
Planner outputs, but their execution route is structured `parametric_update`.

## Question 4: What does C12 sanitizer do?

Read `docs/C12_REPAIR_OUTPUT_SANITIZER.md`, `reports/c12_sanitizer_eval.json`,
and `reports/current_c12_input_manifest.json`. C12 checks current semantic repair
samples for schema validity, oracle copyback, nonmask equality, palette validity,
no-op rows, null categories, allowed-label behavior, and cleanup threshold.

## Question 5: How does EditRoom data become a Qwen/DiffSynth sample?

Read `docs/EDITROOM_TO_SEMANTIC_LAYOUT_CONVERSION_POLICY.md`,
`tools/convert_editroom_pair_to_semantic_layout.py`,
`tools/search_valid_editroom_semantic_repair4_samples.py`, and C12/C14 reports.

Flow:

EditRoom before/after pair -> I_bad -> I_target -> mask_spec -> control_mask ->
repairplan_current.json -> prompt.txt -> metadata.csv.

## Question 6: What are the four DiffSynth metadata fields?

| Field | Meaning | Generated by | Used for | If wrong |
| --- | --- | --- | --- | --- |
| `image` | I_target | dataset packager | training target | model learns wrong target |
| `blockwise_controlnet_image` | I_bad | dataset packager | condition image | condition mismatches task |
| `blockwise_controlnet_inpaint_mask` | control_mask | mask adapter/packager | repaint region | wrong pixels are edited |
| `prompt` | correction_prompt | RepairPlan/packager | text condition | action/category ambiguity |

## Question 7: Where is Qwen/DiffSynth training code?

Read `diffusion/train_diffsynth_qwen_inpaint_lora.sh` for the local template,
`scripts/c13_train_common.sh`, `scripts/c13_train_*.sh`,
`scripts/c14_train_*_20.sh`, and configs under `configs/c13_*` and `configs/c14_*`.

The real server script recorded in reports is
`/wuqingyaoa800/qiuziyan/DiffSynth-Studio/examples/qwen_image/model_training/train.py`.

Explicit parameters include base model Qwen/Qwen-Image, ControlNet
Qwen-Image-Blockwise-ControlNet-Inpaint, batch size 1, learning rate 1e-4,
LoRA rank 32, max_pixels 262144, seed from config/script, and gradient
checkpointing enabled. LoRA alpha/dropout, optimizer, scheduler, and precision
are not explicitly set in local mirrors; treat them as DiffSynth defaults unless
the server script states otherwise.

## Question 8: How was C13 small overfit done?

C13 used 3 rows per action and a 12-row mixed set. It produced 100-step
checkpoints. The result was PARTIAL: data and loader worked, but strict edit
gates were not met. Read `docs/C13_SEMANTIC_REPAIR4_OVERFIT.md` and
`reports/c13_overfit_*.json`.

## Question 9: How was C14 medium diagnostic done?

C14 built 20 rows/action, 80 total. Data gate and loader passed. C14.1 selected a
clear GPU and trained single-action runs to 300 steps plus MIXED_80 to 320 steps.
Sampled image evaluation still has edit_success 0 and high allowed-label
violations. Read `docs/C14_MEDIUM_SCALE_DIAGNOSTIC_TRAINING.md`,
`reports/c14_1_autogpu_resume_result.json`, and
`reports/c14_1_autogpu_image_eval_summary.json`.

## Question 10: How do I read training results?

Look for raw output, snapped output, sanitized output, overlay/contact sheet,
loss curve, and metrics JSON/CSV. Key metrics are masked pixel accuracy, target
IoU, action IoU, object count F1, edit_success, allowed-label violation count,
extra component count, nonmask equality, and palette validity.

## Question 11: Where do I start if I need to modify something?

Read the current state, then the relevant schema, then validator, then tests.
Make the smallest tool/code change, update docs, and run the validator plus
current protocol tests. Never change taxonomy, palette, or routing as a side
effect of a local bug fix.
