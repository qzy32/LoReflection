## 核心定位

> 本版为一致性清理版：全文统一使用 `RepairPlan + mask_spec`；`binary control_mask` 只表示执行层张量；State Observer 固定为程序化回读；VLM 只承担目标构造、语义审查与修正规划；LoState 仅包含 Goal LoState 与 Observed LoState，LoReview、RepairPlan、mask_spec、control_mask、Final3DState 均为 LoState 外部结构。

### 当前冻结版框架

```text
User instruction + Architecture JSON / optional architecture image
        → Target LoState Constructor
        → Goal LoState
        → Qwen-Image initial semantic layout generation
        → Programmatic State Observer
        → Observed LoState
        → Dual-Track State Reviewer + LoRAM
        → LoReview
        → Correction Planner
        → RepairPlan + mask_spec
        → Mask Tensor Adapter
        → binary control_mask
        → Qwen-Image-ControlNet-Inpainting
        → Re-observe / Re-review / AcceptanceController
        → APM lifting
        → Final3DState
```

本版的论文主线是：**one-shot layout generation / editing 缺少生成后验证和局部修正；LoReflection 用统一 LoState 把目标和观测拉到同一状态空间，在 LoReview 和 LoRAM 的问题归因下执行可验收的局部修复，从而把开环生成改造为闭环状态对齐。**


LoReflection 是一个基于 reflective state language（LoState）的室内布局生成与编辑框架。其核心设计哲学是：布局生成模型不应无条件信任自身输出，而应能够将生成结果回读为可检查状态、对照目标规约进行状态审查、在问题区域执行局部修正，并再次验证修正是否生效。框架包含以下七项关键设计：

1. **LoState Reflective State Language**：LoState 不是普通 JSON layout，也不是图神经网络意义上的 graph，而是一种双角色、带关系注释的反思式状态语言。Goal LoState 以 furniture slots、generation hints 和 desired relations 描述“应该满足什么”；Observed LoState 以程序回读得到的 furniture instances、mask、bbox、centroid、measured relations 与 hard-constraint evidence 描述“实际生成了什么”。二者共享同一语义注册表 `semantic_registry_ref` 与建筑引用 `architecture_ref`，使 Qwen-Image 的图像输出能够被翻译为 Qwen2.5-VL 可读、可审查、可规划的状态表达。
2. **Programmatic State Observer**：State Observer 不再设计为 “Layer 1 + VLM Layer 2”。在本文的 fixed-palette top-down semantic layout 设定下，Observer 只做程序化状态回读：颜色色板映射为 label map，连通域分析得到 furniture instances，再计算 bbox、mask、centroid、area、orientation、measured relations 和 hard-constraint evidence。VLM 不覆盖几何真值；语义判断进入后续 State Reviewer。
3. **Dual-Track State Reviewer and LoReview**：Goal 与 Observed 的审查由 Dual-Track State Reviewer 完成。Track A 负责程序化几何审查，包括数量、碰撞、越界、clearance、preservation 等硬约束；Track B 使用 Qwen2.5-VL 进行软语义审查，包括功能关系、动线、文化惯例和空间语义合理性。Reviewer 的输出是 **LoReview**，即结构化状态审查报告，而不是自由文本 judge、单一分数或早期版本中混杂的视觉评论。
4. **LoRAM for Goal–Observed Alignment**：在 LoReview 内部，本文引入 LoRAM（LoState Relation-Aware Matching）对 Goal furniture slots 与 Observed furniture instances 进行关系感知对齐。LoRAM 将对齐形式化为 partial fused relation-aware matching：feature 项刻画类别、建筑锚点、尺寸和追踪兼容性，structure 项刻画 desired relations 在匹配后的满足情况，dummy nodes 直接表示 missing / extra。该模块的目标不是追求全局最优图匹配，而是提供稳定、可解释的问题归因。
5. **Qwen-Image / Qwen2.5-VL 分工明确**：Qwen-Image 是图像生成与编辑模型，用作 layout image generator / Qwen-Image-ControlNet-Inpainting executor；Qwen2.5-VL 是视觉语言理解与状态推理模型，用作 LoState constructor、Track-B semantic reviewer 和 Correction Planner。二者属于同一生态但训练目标与推理接口不同。
6. **Unified VLM Three-Role Design**：主设计采用同一个 Qwen2.5-VL 权重承担 Target LoState Constructor、Track-B Semantic Reviewer、Correction Planner 三种状态相关角色。三者不是三个独立模型，而是同一 VLM 在同一 LoState 语言下通过不同 task prompt 切换任务。
7. **VLM-Guided Mask Planning for Qwen-Image-ControlNet-Inpainting**：Correction Planner 输出 RepairPlan + `mask_spec`。无论修复动作简单还是复杂，mask 的空间语义规划都由 Qwen2.5-VL 完成；程序层只负责将 `instance_ref`、`bbox`、`polygon`、`old_new_union` 等 mask_spec 栅格化为 Qwen-Image-ControlNet-Inpainting 所需的二值 `control_mask`，并进行尺寸、面积和格式合法性检查。

### 针对四个关键问题的最终结论

**Q1：Qwen-Image 和 Qwen2.5-VL 是否是同一个模型？** 不是。本文必须明确写成“两模型协同”：Qwen-Image 负责生成、编辑和 Qwen-Image-ControlNet-Inpainting 局部修复；Qwen2.5-VL 负责视觉理解、LoState 构造、Track-B 语义审查、RepairPlan 与 mask_spec 规划。

**Q2：VLM 三角色是否使用同一个 Qwen2.5-VL 权重？** 主设计建议使用同一个 Qwen2.5-VL 权重，并通过 task prompt / task token 区分三种任务：`[TARGET_STATE_CONSTRUCTION]`、`[SEMANTIC_REVIEW]`、`[CORRECTION_PLANNING]`。这比三个专家模型拼接更有论文价值，因为它把目标构造、状态审查与修正规划统一为同一个 LoState-grounded multimodal reasoner。

**Q3：State Observer 还需要 VLM Layer 2 吗？** 不需要。State Observer 只做程序化状态回读。VLM 不作为 Observer 的第二层，也不修改 bbox、mask、centroid 等几何事实。VLM 的语义判断属于 Dual-Track State Reviewer 的 Track B。

**Q4：mask 到底怎么给？** 本文固定使用 Qwen-Image-ControlNet-Inpainting，要求输入 `control_image + binary control_mask + descriptive final-state prompt`。`control_mask` 的语义规划由 Qwen2.5-VL Planner 输出为结构化 `mask_spec`；程序层只做栅格化、二值化和合法性检查。即：**VLM plans the mask; the system rasterizes and validates it.**

## 1. Introduction

室内家具布局生成与编辑是智能室内设计、虚拟空间构建和交互式场景创作中的基础任务。与一般视觉生成不同，该任务并不只是生成一幅“看起来合理”的布局图，而是需要在具体房间结构和用户需求的共同约束下，得到一组真正可用且空间上合规的家具布置方案。具体而言，系统需要同时考虑房间边界、墙体、门窗、通行区域以及用户自然语言指令，并在此基础上决定家具的类别、数量、位置、朝向及其功能关系。其核心难点在于，视觉合理性并不必然等同于结构合规性：一个布局即使在整体观感上自然，也可能在几何层面出现家具碰撞、越界、遮挡门窗，在功能层面遗漏必要家具或破坏家具间的使用关系，甚至在动线与居住常识层面产生不便或违和。因此，室内家具布局生成与编辑本质上是一个同时涉及语义理解、空间推理、几何约束和居住常识的高度结构化任务。解决这一问题，对于智能室内设计、可控视觉生成以及可验证场景合成均具有重要的学术价值与实际意义。
     然而，现有方法仍主要停留在两种一次性开环生成范式之中。第一类方法将布局生成视为从输入条件到家具配置的单次映射，利用自回归模型或扩散模型直接输出完整布局。这类方法能够捕捉家具间的统计共现与空间关联，却难以保证单次生成结果实际满足碰撞、越界、功能绑定等具体约束，因为统计似然并不等同于约束满足。第二类方法利用大语言模型的常识推理能力生成结构化布局规划，虽然能够输出包含坐标与关系描述的布局方案，但模型并不验证这些输出是否实际满足各项约束，结构化预测同样无法保证约束满足。上述两类方法的共同局限是生成结果可能违反约束，而违反之后系统又缺乏任何发现与纠正的手段：既无法回读生成结果的实际状态，也无法将其与目标约束比对定位违规，更无法针对违规执行局部修正。
      我们的insight在于：一次性开环生成将布局输出视为无需检验的终端结果，但是不经结果与目标的比对，系统便无从确认约束是否真正满足。这意味着布局系统必须将目标规约与生成结果纳入统一的结构化状态表示，使约束差距可观测、可诊断、可修正，而不是在一次性推理后盲目采信输出。为此，本文提出 LoReflection，一个基于 Goal–Observed 状态对齐的闭环反思框架。LoReflection 引入 LoState 作为统一的状态空间，将布局编码为对象、关系、建筑条件与约束状态的结构化组合，使目标与观测共享同一 schema，从而为闭环比对与迭代修正提供可计算的基础。LoState 只描述"应满足什么"而不规定刚性坐标，将精确几何留给生成与回读共同确定。在此基础上，LoReflection 实现迭代式 Generate–Critic–Edit：构造 Goal LoState，生成初始布局后由 State Observer 独立回读为 Observed LoState，比对诊断违规并驱动局部修正，如此迭代直至差距消除。验证由此从跨模态比较转化为结构化差异计算，从而将一次性开环预测重新定义为闭环的状态对齐过程。
     闭环的运转需要一个能够理解空间布局并操作结构化状态的推理器，视觉语言模型兼具图像理解与结构化推理能力，正是这一角色的合理选择。LoReflection 使用同一个 Qwen2.5-VL 在 LoState 的统一约束下承担三重角色：Constructor 从用户指令与建筑条件构造 Goal LoState，Verifier 比对 Goal 与 Observed 诊断约束违规，Planner 将违规转化为局部修正动作。三者都围绕同一 LoState 进行读写与推理：构造是将需求写入状态，诊断是从状态中读取差距，规划是将差距转化为动作。因此，同一 VLM 通过多任务监督微调即可学习在 LoState 约束下切换角色，而无需为每个角色部署独立模型。LoState 以结构化 JSON 编码布局状态，与俯视语义图共同输入 VLM：语义图提供全局空间感知，LoState 提供精确的结构化事实，两者互补使 VLM 既能感知整体布局格局，又能进行精确的约束推理。在执行侧，同一 Qwen-Image 联合 ControlNet 以 inpainting mask 控制重绘范围，mask 覆盖全图即布局生成，仅覆盖目标区域即局部修正，从而以单一模型统一闭环中的生成与编辑。
      最后，现有布局评估主要衡量生成分布与真实分布的统计距离，评估粒度与布局合规的多维性不匹配。为此，本文提出 LoReflection-Eval，一个面向家具布局的细粒度诊断式评估基准，从语义完整性、关系合理性、建筑合规性、几何精确性、功能可用性与编辑保持性六个维度提供可归因的评价，使评估与框架的诊断能力对齐。
1. 提出基于 LoState 统一状态空间的闭环反思框架 LoReflection。我们以同一 schema 同时承载 Goal 规约与 Observed 观测，将验证从跨模态比较转化为结构化差异计算，从而将布局生成与编辑从一次性开环预测重新定义为迭代式 Generate–Critic–Edit 的状态对齐过程。
2. 提出闭环推理与执行的统一模型设计。我们通过多任务监督微调使同一 VLM 在 LoState 约束下承担目标构造、约束验证与修正规划三角色，并以同一生成模型联合 ControlNet 通过可控重绘统一布局生成与局部编辑。
3. 提出面向家具布局的细粒度诊断式评估基准 LoReflection-Eval。我们从语义完整性、关系合理性、建筑合规性、几何精确性、功能可用性与编辑保持性六个维度提供可归因的评价，使评估粒度与布局合规的多维约束对齐。

### 2.2 Target LoState Constructor (Qwen2.5-VL)

**模型**：Qwen2.5-VL-7B（与 Track B Reviewer、Correction Planner 共享同一权重；不同 task prompt）。

**核心判断**：Goal Constructor 可以由同一个 Qwen2.5-VL 承担，而且更适合作为本文主设计。原因不是“VLM 比 LLM 一定更强”，而是本文的目标状态构造并非纯文本任务：输入中既有 Architecture JSON，也有可渲染的户型/语义平面图；目标状态需要把用户意图、门窗位置、房间边界、可通行区域和家具功能关系统一到同一 LoState schema 中。Qwen2.5-VL 具备图像理解、layout/diagram analysis、object grounding 与 JSON 结构化输出能力，因此可作为统一的 multimodal state constructor（Qwen2.5-VL Technical Report, 2025）。LayoutGPT 已经证明 LLM 可将语言与空间关系转化为结构化布局计划，但它主要依赖文本/样例式布局表示；LoReflection 进一步引入可视建筑条件与回读状态，使同一 VLM 能在目标构造、验证和修正规划之间共享空间语义（LayoutGPT, 2023）。

**为什么不建议三个独立微调模型作为主方案？**

| 方案 | 表述强度 | 优点 | 风险 | 建议定位 |
|---|---|---|---|---|
| 同一 Qwen2.5-VL + 不同 prompt | 最强 | 统一状态推理；部署简单；体现 LoState schema 的统一性 | 复杂任务下可能互相干扰 | 主设计、先做 zero-shot/prompt baseline |
| 同一 Qwen2.5-VL + shared LoRA multi-task SFT | 强 | 共享视觉—空间能力，同时适配输出格式 | 需要构造多任务数据 | 推荐实现路径 |
| 同一 backbone + 三个 task-specific LoRA | 中等 | 减少负迁移；仍保留共享 backbone | 创新性弱于完全共享 | 消融/工程 fallback |
| 三个独立 VLM | 弱 | 每个任务可单独优化 | 成本高；像模块拼接；统一性弱 | 只作为 ablation |

因此，论文主张应写成：**we use a single Qwen2.5-VL as a unified LoState-grounded reasoner for target construction, soft verification, and correction planning**。这句话的核心是“同一状态语言 + 同一视觉语言模型 + 不同任务提示”。

### 2.3 Programmatic State Observer：程序化状态回读


### 2.4 LoState as Verification-Enabling State Language

LoState 的设计哲学源于对两个互补学术传统的融合，分别对应 LoState 的**静态结构特性**和**动态执行模式**。

#### 2.4.1 静态结构：Verification-Aware 设计哲学

从 Verification-Aware Intermediate Representation 中，LoState 继承了“表示本身为验证而设计”的核心原则——验证不是事后附加的后处理，而是表示设计的出发点。这一原则在 LoState 中体现为三个具体设计决策：

1. **关系与约束证据的独立性**：Goal LoState 中的 desired relations 由用户需求、建筑条件和 room prior 构造；Observed LoState 中的 measured relations 和 hard-constraint evidence 则由 State Observer 从几何事实中独立计算。这保证了验证的外部性：系统比较的是目标规约与观测事实，而不是生成模型对自身输出的自述。
2. **Goal–Observed 双重角色的结构对称性**：LoState 同时充当 Goal（规格说明：应该满足什么）和 Observed（可观测状态：实际处于什么）两种角色，且两者共享 schema。Goal 使用 furniture slots 和 desired relations；Observed 使用 furniture instances 和 measured relations；LoReview 在二者之间建立 slot-instance 与 relation-level 对齐。
3. **约束的可分解性**：LoState 的结构使约束天然分解为程序可复验的硬约束与需要语义解释的软约束，支撑 Dual-Track State Reviewer。Track A 读取 hard_constraint_evidence，Track B 在 measured relations 和图像上下文上进行语义解释。

#### 2.4.2 动态执行：TOTE 反馈回路与 Observe-Compare-Act 模式

LoReflection 的动态执行模式更精确的理论根基是控制论中的 **TOTE 反馈回路**（Miller, Galanter & Pribram, 1960）：

```text
Goal LoState (G)          ← 目标状态（应该满足什么）
        │
        ▼
┌───────────────┐
│   Test (t)     │  Observed LoState Ŝᵗ vs Goal G
│   比对当前与目标 │  Rᵏ = Review(G, Ŝᵏ)
└───────┬───────┘
        │
    Rᵏ pass? ──── YES → Exit（目标达成）
        │
       NO
        │
        ▼
┌───────────────┐
│   Operate      │  Correction Planner: Rᵏ → RepairPlan aᵗ
│   执行修正操作  │  Inpainting: aᵗ → Iᵗ⁺¹
└───────┬───────┘
        │
        ▼
┌───────────────┐
│   Test (t+1)   │  State Observer: Iᵗ⁺¹ → Ŝᵗ⁺¹
│   再次比对      │  Accept iff C(Ŝᵗ⁺¹) ≺ C(Ŝᵗ)
└───────┬───────┘
        │
    pass or T reached? ──── YES → Exit
        │
       NO → 回到 Operate
```

这一 TOTE 模式在多个学术传统中有对应：

| 学术传统 | 核心思想 | 与 LoReflection 的对应 |
|---------|---------|------------------------|
| TOTE（Miller et al., 1960） | Test-Operate-Test-Exit：目标与当前状态的差距驱动操作 | Goal LoState = Test reference, Observed = current state, Correction = Operate |
| Hindsight Experience Replay（Andrychowicz et al., 2017） | achieved state vs goal state 比较驱动学习 | LoReflection 将比较结果用于驱动修正（推理时），而非策略学习（训练时） |
| Reveal-to-Revise（Pahde et al., MICCAI 2023） | Reveal（解释输出）→ Revise（修正）→ Verify | State Observer = Reveal, Correction Planner + Inpainting = Revise, Re-verify = Verify |
| Generate-but-Verify | 生成后验证，外部验证信号驱动修正 | Dual-Track State Reviewer 提供外部验证信号，不是模型自纠错 |

#### 2.4.3 LoState 的核心定位：Verification-Enabling State Language

综合上述分析，LoState 的准确定位不是单纯的后验观察语言，而是一种 dual-role verification-enabling state language：它同时承担生成前的目标规约和生成后的状态观测。Goal LoState 在生成前由用户指令、Architecture JSON 和可选建筑布局图构造而来，用于描述当前任务中“应该满足什么”，包括家具槽位、数量要求、生成提示、期望关系、建筑禁区和编辑保持目标；Observed LoState 则在生成后由 State Observer 从 Qwen-Image 输出的俯视语义布局图中回读得到，用于描述模型“实际生成了什么”，包括家具实例、bbox/mask、几何测量、碰撞、越界、clearance 遮挡和软约束证据。

因此，LoState 的关键创新不在于单纯从生成流程后端介入，而在于用同一套结构化状态语言打通“目标构造—图像生成—状态回读—约束验证—局部修正”整个闭环。在生成前，Goal LoState 将自然语言需求和建筑条件转化为可执行的布局规格，并进一步被 Prompt / Condition Builder 转化为 Qwen-Image 的生成条件；在生成后，Observed LoState 将不可直接验证的图像输出重新转化为可比较、可计算、可审计的结构化状态。验证过程由 LoReview 驱动：对象级差距用于检测家具缺失或多余，关系级差距用于检测功能绑定失败，建筑级和几何级差距用于定位碰撞、越界、门窗遮挡和非目标漂移。

这一设计与“先输出 LoState 再渲染”的纯前端布局生成方法不同。LoReflection 并不要求底层生成模型直接生成最终 LoState，也不依赖一个确定性 renderer 将 LoState 渲染成图像；相反，Qwen-Image 仍然作为独立的图像生成/编辑模型工作，LoState 只在其外部提供目标规约、观测回读和验证反馈。换言之，LoState 不是替代 Qwen-Image 的生成表示，而是包裹在 Qwen-Image 外部的状态接口：前端负责约束生成方向，后端负责检查实际结果，并在两者不一致时驱动局部修正。

这种 dual-role 设计带来两个优势。第一，它保留了图像生成模型的开放生成能力，不把布局生成压缩成刚性的参数渲染问题；第二，它又避免了端到端图像输出不可验证的问题，因为每次生成结果都会被回读为 Observed LoState，并与 Goal LoState 进行结构化比对。由此，LoReflection 将一次性开环生成转化为“目标规约—生成—观察—比对—修正—再观察”的闭环系统。

### 2.5 Dual-Track State Reviewer

Dual-Track State Reviewer 读取 Goal LoState、Observed LoState、当前语义布局图像和 Architecture JSON，输出 LoReview。LoReview 是结构化状态审查报告，负责汇总几何违规、语义问题、观测问题、slot-instance 对齐结果、问题严重度和修复建议。

#### Track A：程序化几何审查

Track A 直接从 Observed LoState 和 Architecture JSON 计算，负责：

```text
1. count match；
2. collision；
3. within boundary；
4. clearance overlap；
5. non-target preservation；
6. required relation 的几何可验证部分。
```

Track A 不需要训练；在给定 State Observer 回读几何的条件下，验证过程是确定性、可复验的。若 parser confidence 低且影响 required slot、protected instance 或 hard evidence，则不让 VLM 猜测，而是生成 `observation_issue` 并触发 REOBSERVE。

#### Track B：Qwen2.5-VL 语义审查

Track B 的输入不是裸图，而是：

```text
current semantic layout image
+ Goal LoState
+ Observed LoState
+ Track A review summary
+ LoRAM ambiguous cases
```

Track B 负责解释规则难以穷举的软语义问题，例如功能关系、动线可用性、文化惯例、视觉平衡和 same-category ambiguity。它不覆盖程序化几何事实，也不修改 Observed LoState。

#### LoReview 输出

LoReview 包含：

```text
1. LoRAM slot-instance alignment；
2. entity_missing / entity_extra；
3. category_mismatch；
4. relation_issue / soft_relation_issue；
5. hard_geometry_issue；
6. preservation_issue；
7. observation_issue；
8. ambiguous_alignment；
9. recommended_next_step。
```

### 2.6 LoRAM：Partial Fused Relation-Aware Matching

在 LoReview 判断布局问题之前，Reviewer 必须先建立 Goal furniture slots 与 Observed furniture instances 的对齐关系。这个过程不是简单的类别计数：一个 observed instance 可能以不完全相同的类别满足目标 slot 的空间功能角色，例如床边的 `small_table` 可能应被解释为错误类别的 `nightstand`；多个同类别实例也可能分别对应不同的目标关系。因此，本文提出 **LoRAM: LoState Relation-Aware Matching**，作为 LoReview 内部的关系感知对齐模块。

#### 2.6.1 Partial fused matching 目标函数

给定 Goal slots $S=\{s_i\}$、Observed instances $O=\{o_j\}$ 和 Goal desired relation set $E_G$，LoRAM 将对齐形式化为 partial fused relation-aware matching：

```text
E(X)
= (1 - α) · Ẽ_feature(X)
+ α · Ẽ_structure(X)
+ E_penalty(X)
```

其中：

```text
Ẽ_feature(X)
= 1 / |S| · Σ_{i∈S,j∈O} X_ij D^F_ij
```

```text
D^F_ij
= λ_cat C_cat(s_i,o_j)
+ λ_anchor C_anchor(s_i,o_j)
+ λ_size C_size(s_i,o_j)
+ λ_track C_track(s_i,o_j)
```

```text
Ẽ_structure(X)
= 1 / max(|E_G|,1) ·
  Σ_{(s_i,p,s_k)∈E_G} Σ_{j∈O,l∈O}
  X_ij X_kl D^R_{ik,jl}(p)
```

```text
E_penalty(X)
= Σ_i X_{i,∅} μ_miss(s_i)
+ Σ_j X_{∅,j} μ_extra(o_j)
```

约束为：

```text
Σ_j X_ij ≤ 1,
Σ_i X_ij ≤ 1,
X_ij ∈ {0,1}.
```

这里，$D^F$ 是 slot-instance 的 feature distance，包含类别兼容性、建筑锚点一致性、尺寸先验和 repair-round tracking；$D^R$ 是 relation structure distance，衡量当 $s_i$ 与 $s_k$ 分别匹配到 $o_j$ 与 $o_l$ 时，Observed 中是否满足 Goal relation predicate $p$。结构项只对 $E_G$ 中显式声明的 desired relations 计算，而不是对所有 object pairs 计算，以避免 dense relation 噪声。

该形式自然结合了 feature consistency 与 structure consistency，类似 Fused Gromov-Wasserstein alignment 的思想，但 LoRAM 与通用 FGW 不同：它输出离散 assignment 和明确的 missing / extra / mismatch 归因，而不是 soft transport plan；它只使用 Goal-specified sparse relations；并且把语义歧义交给 Qwen2.5-VL reviewer，而不是由匹配算法单独决定。

#### 2.6.2 Relation cost 的硬软区分

Goal relation 中的 `necessity` 决定 $D^R$ 的代价类型：

```text
if necessity == required:
    D^R = hard_violation_cost
else if necessity == preferred:
    D^R = soft_violation_cost
```

其中：

```text
hard_violation_cost = 0 if satisfied, else M_hard
soft_violation_cost = β · violation²
```

`M_hard` 是较大的有限值，而不是无穷大。类别不兼容的 pair 仍由 semantic gating 先排除，避免系统为了满足关系而做出荒唐类别匹配。

#### 2.6.3 求解：Iterative Hungarian Refinement

完整目标含二次结构项，直接求解没有必要。LoRAM 对该目标进行轻量线性化近似：

```text
Phase 1: Cost Construction
  1. 通过 semantic_registry_ref 做 candidate gating；
  2. 计算 D^F；
  3. 对 incompatible pairs 设大代价 M；
  4. 加入 dummy row/column 表示 missing 与 extra。

Phase 2: Iterative Relation-Aware Assignment
  1. π⁰ = Hungarian(D^F with dummy nodes)；
  2. 对每轮 k，计算局部平均 relation-induced cost C̄_rel^k(i,j)；
  3. C_eff^k(i,j) = (1-α)D^F_ij + αC̄_rel^k(i,j)；
  4. πᵏ⁺¹ = Hungarian(C_eff^k)；
  5. 若 E(πᵏ⁺¹) > E(πᵏ)，回退到 πᵏ 并停止；
  6. 若 πᵏ⁺¹ = πᵏ 或达到 K 次迭代，停止。
```

计算 $C̄_{rel}^k(i,j)$ 时，LoRAM 临时假设 $s_i$ 匹配到 $o_j$，其他 slots 保持当前 assignment $π^k$。这种 tentative assignment 只用于估计关系诱导代价，可能与当前匹配冲突；随后的 Hungarian 步骤会重新恢复一对一约束。

如果某条 relation 的另一端 slot 当前未匹配，则该 relation 被标记为 `relation_untestable`，不向已匹配端点添加惩罚。真正的问题由 unmatched endpoint 产生，例如 `entity_missing: slot_nightstand_1`，而不是错误惩罚已经正确匹配的 bed。

#### 2.6.4 Ambiguity Review 与 issue 映射

LoRAM 在以下场景触发 Track-B Qwen2.5-VL review：

```text
Type A: top-1 / top-2 assignment cost gap < δ_amb；
Type B: category evidence 与 relation evidence 冲突；
Type C: parser confidence 低且影响 required/hard/protected evidence → REOBSERVE；
Type D: same-category multi-instance ambiguity。
```

Type C 不交给 VLM 猜测，而是触发 re-observation 或 parser refinement。

LoRAM 输出到 LoReview 的映射如下：

| LoRAM 结果 | LoReview issue | Planner action |
|---|---|---|
| required slot → ∅ | entity_missing | INSERT |
| optional slot → ∅ | optional_entity_missing | optional INSERT / ignore |
| instance → ∅ | entity_extra | DELETE |
| exact match, all relations consistent | no issue | no action |
| substitutable category match | category_mismatch | REPLACE |
| required relation violated | relation_issue | MOVE / ADJUST |
| preferred relation violated | soft_relation_issue | MOVE lower priority |
| relation endpoint unmatched | relation_untestable + entity_missing | INSERT first |
| parser confidence affects hard evidence | observation_issue | REOBSERVE |
| same-category ambiguity | ambiguous_alignment | VLM review |

LoRAM 的价值不在于平均 alignment accuracy 一定大幅提升，而在于减少灾难性误归因和不必要编辑。例如，它应避免把“床边的小桌子应替换为床头柜”误判为“缺失床头柜 + 多余小桌子”，从而降低 unnecessary edit rate。

### 2.7 LoRepair-SD：LoReview 驱动的局部修复算法

LoRepair-SD（State-Review-guided Local Repair）是本文将反思闭环落到执行层的主算法。其核心不是训练新的图模型，也不是让 VLM 自由修图，而是将 LoReview 中的可归因问题转化为 RepairPlan 与 VLM mask_spec，再由 Qwen-Image-ControlNet-Inpainting 执行局部修复，最后重新观察并验收。

```text
LoRepair-SD(G, Iᵏ, Ŝᵏ, Rᵏ, A):
  1. Rank issues in LoReview Rᵏ by severity, locality, fixability and preservation risk.
  2. Select the target issue vᵏ.
  3. Use Qwen2.5-VL Planner to map vᵏ to RepairPlan aᵏ and mask_spec mspecᵏ.
  4. Rasterize mspecᵏ into binary control_mask Mᵏ using Mask Tensor Adapter.
  5. Run Qwen-Image-ControlNet-Inpainting with Iᵏ, Mᵏ and descriptive prompt Pᵏ.
  6. Re-observe output image into Ŝᵏ⁺¹ and generate LoReview Rᵏ⁺¹.
  7. Accept if hard constraints do not regress and LoReview improves; otherwise rollback.
```

一次反射循环只优先修复一个或少数强局部问题，避免一次编辑覆盖过大区域而破坏已正确对象。Violation Ranker 对 LoReview 中的问题进行排序：

```text
score(v)
= α_sev severity(v)
+ α_loc locality(v)
+ α_fix fixability(v)
- α_risk preservation_risk(v)
- α_cost estimated_edit_area(v)
```

#### LoRepair-SD 与图一致性修复的边界

LoState 可以包含 relation-annotated edges，也可以按需投影为轻量关系视图，但本文不将其定义为 LoStateGraph，也不把主方法建立在 GNN、graph diffusion 或 message passing 上。LoRAM 使用 partial fused matching 只是为了 LoReview 问题归因，而不是把论文主贡献转向 graph matching solver。图一致性传播修复可作为未来扩展，但不作为主线。

### 2.8 Correction Planner：同一 Qwen2.5-VL 的第三角色

Correction Planner 是同一 Qwen2.5-VL 的第三个任务角色。它读取 Goal LoState、Observed LoState、LoReview、当前语义布局图像和目标 issue，输出 RepairPlan 与结构化 `mask_spec`。Planner 不直接输出最终 PNG mask，也不把 bbox/mask 决策交给程序主导；它负责 mask 的空间语义规划，程序层只负责栅格化与合法性检查。

**输入**：

```text
[CORRECTION_PLANNING]
Goal LoState G
Observed LoState Ŝᵏ
LoReview Rᵏ
Target issue vᵏ
Current semantic layout image Iᵏ
Architecture summary A
```

**输出（RepairPlan + mask_spec）**：

```json
{
  "action_id": "action_round0",
  "action_type": "REPLACE",
  "target_ref": {
    "matched_slot": "slot_nightstand_1",
    "instance_id": "small_table_1",
    "target_category": "nightstand"
  },
  "trigger_issue": "issue_replace_small_table_as_nightstand",
  "spatial_reasoning": {
    "why_this_action": "small_table_1 satisfies the bedside spatial role but has a substitutable wrong category.",
    "preserve_refs": ["bed_1", "arch:door_001_clearance"]
  },
  "mask_spec": {
    "generated_by": "Qwen2.5-VL_CorrectionPlanner",
    "mask_type": "instance_ref",
    "instance_refs": ["small_table_1"],
    "coordinate_space": "image_pixel",
    "image_size_px": [1024, 1024],
    "binary_value": {"edit_region": 255, "preserve_region": 0},
    "postprocess": {
      "dilate_px": 8,
      "min_area_px": 512,
      "max_area_ratio": 0.18
    }
  },
  "protected_refs": ["bed_1", "arch:door_001_clearance"],
  "correction_prompt": "A top-down fixed-palette semantic bedroom layout where the masked object is a nightstand beside the bed. The bed, walls, door clearance, and all other furniture remain unchanged.",
  "acceptance_criteria": [
    "target_category_is_nightstand",
    "adjacent_to_bed",
    "no_collision",
    "no_clearance_overlap",
    "non_target_preservation"
  ]
}
```

`correction_prompt` 应描述修复后的最终状态，而不是使用 “delete / move / fix” 等操作式命令。Qwen-Image-ControlNet-Inpainting 的接口更适合 descriptive final-state prompt，因此 RepairPlan 中的 action_type 用于系统审计与训练监督，而输入执行器的 prompt 应描述整幅语义布局及 masked region 的目标状态。

### 2.9 Spatial Context Provider（可选）

早期版本中的 Spatial Context Provider 容易让读者误解为程序主导 mask 区域选择。v1.4 后，主路径改为 VLM-guided mask planning：Qwen2.5-VL Planner 始终输出 mask_spec。Spatial Context Provider 只作为可选辅助，为 VLM 提供空间摘要，而不替 VLM 决定编辑区域。

它可以提供：

```text
1. free-floor summary；
2. architecture clearance regions；
3. protected object refs；
4. distance-to-anchor statistics；
5. collision / overlap evidence；
6. candidate empty regions as optional context。
```

这些信息进入 Planner prompt，帮助 VLM 进行空间语义规划。Planner 可以使用这些摘要，但最终仍必须显式输出 `mask_spec`。

### 2.10 VLM-Guided Mask Planning and Mask Tensor Adapter

#### 2.10.1 mask_spec 与 control_mask 的区别

`mask_spec` 是 Qwen2.5-VL Planner 输出的结构化编辑区域计划；`control_mask` 是 Qwen-Image-ControlNet-Inpainting 的执行层输入张量。二者不能混淆。

```text
mask_spec：语义规划，由 VLM 输出
control_mask：二值灰度 mask，由程序栅格化得到
```

#### 2.10.2 支持的 mask_spec 类型

```text
1. instance_ref：适合 DELETE / REPLACE / ADJUST_ORIENTATION；
2. bbox：适合规则局部 INSERT / MOVE 新位置；
3. polygon：适合不规则插入区域；
4. old_new_union：适合 MOVE，覆盖旧位置和新位置；
5. full_regenerate：仅作为局部修复失败后的 fallback。
```

MOVE 必须使用 `old_new_union`，因为 inpainting 实际上需要同时清除旧位置并在新位置生成目标对象。

#### 2.10.3 Mask Tensor Adapter

Mask Tensor Adapter 不做语义决策，只负责：

```text
1. decode instance_ref / rasterize bbox / polygon / old_new_union；
2. convert to single-channel grayscale mask；
3. binarize to {0,255}；
4. ensure same size as control_image；
5. ensure width/height divisible by 16 if required；
6. check min_area / max_area_ratio；
7. return INVALID_MASK_PLAN if invalid。
```

如果 mask_spec 非法，系统不应悄悄改成另一个区域，而应要求 Planner 重新规划。这样保证 VLM 仍是 mask 空间语义决策者。

#### 2.10.4 Qwen-Image-ControlNet-Inpainting 输入规范

本文固定使用 Qwen-Image-ControlNet-Inpainting 作为局部修复 executor。其输入为：

```text
control_image = current semantic layout image Iᵏ
control_mask  = binary grayscale mask Mᵏ
prompt        = descriptive final-state correction prompt Pᵏ
```

mask 规范：

```text
white / 255 = edit region
black / 0   = preserve region
```

默认不使用 soft feather 作为主路径，因为该类 pipeline 会将 mask 转灰度并二值化。二值 dilation、面积检查和连通域清理可以作为 Adapter 的后处理，但不改变 VLM 的语义决策。

### 2.11 Qwen-Image-ControlNet-Inpainting Correction

局部修复执行器固定为 Qwen-Image-ControlNet-Inpainting。本文不把修复执行器写成泛化黑盒，因为训练与推理都依赖明确的输入格式：

```text
Iᵏ + binary control_mask Mᵏ + descriptive prompt Pᵏ → Iᵏ⁺¹
```

其中 `Mᵏ` 由 Mask Tensor Adapter 从 VLM 的 `mask_spec` 栅格化得到。Prompt 不描述“执行什么操作”，而描述“修复后图像应是什么状态”。例如：

```text
Bad: Delete the extra chair.
Good: A top-down fixed-palette semantic bedroom layout where the masked region is empty floor and all existing furniture outside the mask remains unchanged.
```

执行完成后，输出图像必须重新进入 Programmatic State Observer。修复是否成功不由执行器自述决定，而由新的 Observed LoState 与 LoReview 审查决定。

### 2.12 Revision Controller & Re-verify

Revision Controller 负责记录每轮状态、执行接受/回滚，并决定是否继续迭代。每轮 accepted repair_round 应写入 repair_round log：

```json
{
  "task_id": "bedroom_case_001",
  "repair_round": 1,
  "previous_observed_state_id": "obs_bedroom_case_001_round0",
  "candidate_observed_state_id": "obs_candidate_t1",
  "accepted_observed_state_id": "obs_t1",
  "action_id": "act_t0_insert_nightstand",
  "accepted": true,
  "cost_before": [1, 1, 0, 0, 0.12],
  "cost_after": [0, 0, 0, 0, 0.08],
  "new_hard_errors": [],
  "rollback_reason": null
}
```

接受准则采用字典序成本下降，并显式禁止新硬错误和非目标退化：

```text
C(Ŝᵗ) = (
  N_hard(Ŝᵗ),
  N_required_relation(Ŝᵗ),
  N_preservation(Ŝᵗ),
  N_soft(Ŝᵗ),
  U_observation(Ŝᵗ)
)

Accept(I') iff
  C(Ŝ') ≺ C(Ŝᵗ)
  and no_new_hard_violation(Ŝ', Ŝᵗ)
  and no_non_target_regression(Ŝ', Ŝᵗ)
```

**算法伪代码**：
```text
Algorithm: LoRepair-SD Reflective Layout Correction
Input:  Goal state G, initial layout I⁰, Architecture A, max iterations T
Output: Corrected layout I* or best-effort layout

1:  Ŝ⁰ ← Observe(I⁰, A)
2:  for t = 0, 1, ..., T-1 do
3:    Rᵏ ← Compare(G, Ŝᵗ, A)                         // LoReview
4:    if Pass(Rᵏ) then return Iᵗ end if
5:    vᵗ ← RankAndSelect(Rᵏ)                          // Violation Ranker
6:    aᵗ ← PlanCorrection(vᵗ, Rᵏ, Iᵗ, Ŝᵗ, G)          // Qwen2.5-VL Planner
7:    Rᵗ ← GenerateCandidateRegions(aᵗ, Ŝᵗ, A)        // Programmatic candidates
8:    Mᵗ, Pᵗ ← ConstructMaskPrompt(aᵗ, Rᵗ, Ŝᵗ, A)    // Programmatic mask
9:    I' ← Inpaint(Iᵗ, Mᵗ, Pᵗ)                        // Qwen-Image / ControlNet
10:   Ŝ' ← Observe(I', A)                              // Re-observe
11:   if Accept(Ŝ', Ŝᵗ) then
12:      Iᵗ⁺¹ ← I'; Ŝᵗ⁺¹ ← Ŝ'
13:   else
14:      Iᵗ⁺¹ ← Iᵗ; Ŝᵗ⁺¹ ← Ŝᵗ                         // rollback
15:   end if
16: end for
17: return Iᵀ
```

上述算法包含三层误差屏障：(1) **观测不确定性屏障**——低置信度回读结果标记为 uncertainty，优先 REOBSERVE 或降低 destructive edit 优先级；(2) **候选区域屏障**——Planner 不能自由画 mask，只能使用程序化候选区域；(3) **接受准则屏障**——若修正没有降低验证成本或引入新硬错误，则回滚到上一轮布局。三层屏障共同抑制误差级联。

### 2.13 APM 与 Vertical Gate-Check

**语义布局闭环通过后**，APM 一次性恢复 3D 属性：

```text
height:   from category prior + size regression
z_offset: from anchor + category
anchor:   wall / floor / ceiling (from category + position rules)
retrieved_model_id: from category + size + style prior
stability estimation
```

APM 之后执行一次 Vertical Gate-Check（非迭代）：
1. 所有家具的 anchor 是否合法；
2. z_offset 是否导致穿层；
3. height 是否与 category prior 严重偏离；
4. stability estimation 是否通过；
5. retrieved_model_id 是否与类别和尺寸兼容。

关键：LoReflection 的反思闭环运行在俯视语义布局及其 LoState 上；APM 只在布局通过验证后将其提升为 Final 3D Layout State，并执行一次性 vertical consistency check。APM 不参与迭代验证循环，因为图像 inpainting 主要修正平面语义布局，无法直接调整 height、z_offset 或具体 3D 模型。

## 3. 问题形式化定义

### 3.1 输入与输出

**输入**：
- 房间建筑条件 $A$：boundary, wall, door, window, clearance region, coordinate transform；
- 用户需求 $U$：自然语言描述；
- 默认验证配置 $\Pi$：count match, collision, boundary, clearance, preservation 等检查项；
- 可选编辑输入 $I_{\text{init}}$：已有布局图像及其历史状态。

**输出**：
- 布局图像 $I^*$：满足目标规约与约束的俯视语义布局图；
- 最终观测状态 $\hat{S}^*$：通过 State Observer 回读得到的 Observed LoState；
- 修正轨迹 $\{R^{(k)}, v^{(t)}, a^{(t)}, R^{(t)}\}$：每轮 LoReview、目标违规、RepairPlan 与候选修复区域；
- Final 3D Layout State：APM 恢复后的 height、z_offset、anchor、retrieved_model_id 与 vertical check 结果。

### 3.2 反射式生成目标

$$G = \text{ConstructGoal}(U, A, \Pi)$$

$$I^{(0)} = \text{Generate}(G, A)$$

反射循环：

$$\hat{S}^{(t)} = O(I^{(t)}, A)$$

$$R^{(k)} = \text{Compare}(G, \hat{S}^{(t)}, A)$$

$$R^{(k)} = R^{(k)}_A \cup R^{(k)}_B$$

$$\text{if } \text{Pass}(R^{(k)}) \text{ then exit}$$

$$a^{(t)} = \text{CorrectionPlanner}(R^{(k)}, I^{(t)}, \hat{S}^{(t)}, G)$$

$$M^{(t)}, P^{(t)} = \text{MaskTensorAdapter}(a^{(t)}, \hat{S}^{(t)}, A)$$

$$I^{(t+1)} = \text{Inpaint}(I^{(t)}, M^{(t)}, P^{(t)})$$

其中 $O$ 为 State Observer，$D_A$ 为 Track A 的确定性几何差异，$D_B$ 为 Track B 的语义约束差异。$v^{(t)}$ 是当前轮被选择的目标违规，$a^{(t)}$ 是 RepairPlan，$R^{(t)}$ 是程序化候选修复区域。RepairPlan 不是最终 mask；最终 mask 由程序根据 RepairPlan、候选区域、Observed LoState 和 Architecture JSON 构造。

### 3.3 接受准则

$$\text{Accept } I^{(t+1)} \iff C(\hat{S}^{(t+1)}) \prec C(\hat{S}^{(t)}) \wedge \neg\text{new\_hard\_error} \wedge \neg\text{target\_failure} \wedge \neg\text{non\_target\_regression}$$

---

## 4. 训练方案

### 4.1 需要训练的组件

| 组件 | 是否需要训练 | 当前建议 | 训练 / 实现方式 |
|------|------------|----------|----------------|
| Target LoState Constructor (Qwen2.5-VL) | 推荐 | 主方法纳入 shared multi-task LoRA | 由用户指令、Architecture JSON / 平面图、GT scene 反推 Goal LoState，监督 JSON 输出 |
| Programmatic State Observer | **不需要** | 冻结 | 固定色板映射、连通域、bbox/mask/centroid/orientation、collision/OOB/door-window blocking、measured relations |
| Track A Reviewer | **不需要** | 冻结 | 程序化硬约束检查：count、collision、OOB、door/window blocking、clearance、preservation |
| Track B Reviewer (same Qwen2.5-VL) | 需要 | 与 Constructor / Planner 共享 backbone 与 LoRA | 监督输出 soft semantic issues、functional relation issues、uncertainty 与 evidence refs |
| LoRAM | **不需要端到端训练** | 冻结 + 可调超参 | partial fused relation-aware matching；动态挖掘 wrong-category candidates；输出 slot-instance alignment 与 missing/extra/category_mismatch 归因 |
| Correction Planner (same Qwen2.5-VL) | 需要 | 主训练对象之一 | 输入 Goal / Observed / LoReview / 当前图像 / architecture summary，输出 RepairPlan + `mask_spec` + descriptive final-state prompt |
| Mask Tensor Adapter | **不需要** | 冻结 | 只把 `mask_spec` 栅格化为 binary `control_mask`，做合法性检查；不重新决定语义编辑区域 |
| AcceptanceController | **不需要** | 冻结 | 字典序 cost、no-new-hard-error、non-target preservation、edited-area compatibility、rollback |
| Qwen-Image-ControlNet-Inpainting | 建议做领域适配 | 初期可用预训练，论文主结果建议 LoRA | 训练 image + binary control_mask + descriptive prompt → target semantic layout，用于 fixed-palette layout repair |

**路径冻结**：State Observer 不设置 VLM 几何回读层；程序层不主导语义 mask 选择，只执行 `mask_spec → binary control_mask` 的 Adapter；开放式分割工具不进入 fixed-palette semantic layout 主路径；RepairPlan、`mask_spec`、`control_mask` 与 Final3DState 均不属于 LoState 本体。

### 4.2 训练数据构造

#### 数据来源
- **3D-FRONT / 3D-FUTURE**：用于提取 Architecture JSON、家具类别、俯视语义布局、家具属性以及 room-conditioned category / relation priors；
- **Structured3D / 其他室内场景数据**：作为建筑结构和语义标注补充；
- **合成扰动数据**：在正确布局上随机制造缺失、碰撞、越界、clearance 遮挡、关系破坏和非目标漂移，自动生成 LoReview 与 RepairPlan 标注。

#### Goal Construction 数据

从 GT layout 中构造伪用户需求和 Goal LoState：

```text
GT furniture instances + Architecture JSON
  → furniture_slots
  → desired_relations mined from room-conditioned priors
  → verification_profile
```

L2 room prior 不作为硬规则，而是数据驱动软先验：
- 当用户需求模糊时，prior 可推荐常见家具类别；
- 当用户已明确家具类别时，prior 可补充关系先验，例如 nightstand adjacent_to bed；
- 当用户明确排除某类家具时，prior 不能越权添加该家具。

#### Track B 验证数据

自动标注方案：
```python
for scene in dataset:
    observed = observe(scene.semantic_layout, scene.architecture_json)
    goal = construct_goal(scene.instruction, scene.architecture_json)

    # 规则1：床与门的方向/视线关系 → cultural violation candidate
    if bed_directly_faces_door(observed, scene.architecture_json):
        label_soft_violation("cultural_convention", "bed_faces_door", evidence="geometry_then_semantic")

    # 规则2：书桌远离窗户 → functional / lighting violation
    if desk_far_from_window(observed, threshold=1.2):
        label_soft_violation("lighting_preference", "desk_not_near_window", evidence="distance")

    # 规则3：动线阻挡 → accessibility violation
    if blocks_navigation_path(observed, scene.architecture_json):
        label_soft_violation("accessibility", "blocked_path", evidence="free_space_graph")
```

#### Correction Planner / LoRepair-SD 数据

对每个 violation 构造修正轨迹，而不是只构造单步 action：

```text
GT layout
  → perturbation
  → corrupted layout Iᵗ
  → observed corrupted state Ŝᵗ
  → LoReview Rᵏ
  → ranked violation vᵗ
  → RepairPlan aᵗ
  → candidate regions Rᵗ
  → mask / prompt Mᵗ, Pᵗ
  → repaired candidate I'
  → accept / reject label
```

自动修正规则用于生成初始监督信号：missing → INSERT，extra → DELETE，collision / relation_violation → MOVE，wrong_category → REPLACE，low_confidence_observation → REOBSERVE。随后使用 Observer + Reviewer 自动判断修正是否成功，形成可选偏好数据或 GRPO reward。

训练样本格式：
```json
{
  "id": "sample_001",
  "image": "scene_042_semantic.png",
  "task": "CORRECTION_PLANNING",
  "goal_lostate": "{...}",
  "observed_lostate": "{...}",
  "lodiff": "{...}",
  "candidate_regions": [
```

### 4.3 训练阶段

#### Stage 1：Multi-Task SFT

- **数据**：Target Construction 数据 + Soft Verification 数据 + Correction Planning 数据，建议 1:1:1 或按任务难度加权混合；
- **方法**：标准 SFT，LM loss（next-token prediction）；
- **LoRA 配置**：rank=64, alpha=128, target=all linear layers；
- **超参**：lr=1e-5, batch_size=16, 3 epochs；
- **关键**：同一 Qwen2.5-VL 权重，不同 task prompt / task token 区分三种角色，模型学习在同一 LoState schema 下切换输出格式。

```text
[TARGET_STATE_CONSTRUCTION] → VLM 输出 Goal LoState JSON
[SOFT_VERIFICATION]         → VLM 输出 soft violations + evidence objects / region hints
[CORRECTION_PLANNING]       → VLM 输出 RepairPlan + correction prompt
```

#### Stage 2（可选）：GRPO 精修

- **目标**：让 VLM 的软验证和修正建议更有效；
- **Reward 设计**：
  - $r_{verify}$ = Track B 验证结果与人工/规则标注的 F1；
  - $r_{action}$ = RepairPlan 的动作类型、target、protected_refs、mask_spec 是否合理；
  - $r_{correction}$ = 执行修正后 violation 是否被修复（1/0）；
  - $r_{preserve}$ = 非目标对象是否保持；
  - $r_{total} = \alpha r_{verify} + \beta r_{action} + \gamma r_{correction} + \delta r_{preserve}$。
- **GRPO 流程**：对每个输入采样 K=4 个输出，计算 reward，用 group-relative advantage 更新策略；
- **超参**：lr=5e-7, 1 epoch。

## 5. 实验设计

### 5.1 实验要证明什么

实验主线只回答一个核心问题：现有方法大多是 one-shot generation / one-shot editing，而 LoReflection 是否能通过 observe → review → repair → verify 的闭环，把初始布局进一步提升为更高质量、更可用、更少硬错误的最终 3D scene。主文实验应围绕三条证据链展开：

1. **端到端质量**：Ours-Final 是否优于 one-shot baselines 与 Ours-Initial；
2. **闭环机制有效性**：反射轮次、LoState / LoReview / LoRAM、mask_spec、AcceptanceController 是否分别带来可解释收益；
3. **可用性与成本**：Practical Usability 是否补足传统指标盲区，且 2–3 轮反思是否能在可接受成本内收敛。

### 5.2 Dataset and Benchmark Package

主数据来源固定为 **3D-FRONT / 3D-FUTURE**。每个评估样本构造成 LoReflection-Eval benchmark package：

```text
S_i = {
  A_i: architecture annotation,
  Y_i: ground-truth object layout,

```

这里 `S_i` 不是要求所有方法接收的统一输入协议，而是同一底层场景的多视图评估包。不同 baseline 仍可使用各自 native input protocol；评估时统一转换到 method-agnostic evaluation representation 后再计算指标。

- `A_i`：room type、room boundary、door / window anchors、coordinate transform；
- `Y_i`：GT object layout，包括 category、center、size、orientation、footprint、optional asset reference；
- `P_i`：coarse / medium / fine 三种文本粒度。coarse 只描述房间类型和主要家具；medium 加入常见空间关系；fine 加入方向、距离、门窗和 clearance 约束；
- `C_i`：评估约束规格，包括 required_objects、required_relations、functional_relations、geometric_rules、usability_rules。

2. **Architecture JSON 生成器**：从 3D-FRONT 提取 boundary、door/window anchors、clearance regions、coordinate transform 的脚本必须稳定；
3. **Evaluation representation converter**：所有 baseline 输出必须能转成统一 representation；

5. **LoReview schema**：Track A / Track B 输出字段、issue taxonomy、evidence_refs、severity 必须冻结；
6. **LoRAM 实现**：slot-instance matching、dummy nodes、dynamic wrong-category candidate mining、ambiguity review 必须可复现；
7. **RepairPlan + mask_spec schema**：`instance_ref`、`bbox`、`polygon`、`old_new_union`、postprocess、protected_refs、acceptance_criteria 必须定稿；
8. **Mask Tensor Adapter**：mask_spec → binary control_mask 的栅格化、面积阈值、非法 mask feedback 必须完成；
9. **Perturbation generator**：missing、extra、misplaced、collision、OOB、wrong relation、wrong category、non-target drift 的扰动数据生成脚本必须完成；
10. **Qwen-Image-ControlNet-Inpainting 数据对**：`I_bad + control_mask + descriptive prompt → I_gt` 的训练三元组必须从同一语义色板生成；
11. **AcceptanceController**：字典序 cost、no-new-hard-error、non-target preservation、rollback log 必须先实现；
12. **Baseline protocol**：ATISS / DiffuScene / SemLayoutDiff / DirectLayout / ReSpace / editing baselines 的输入输出协议、可复现性和缺失项处理策略必须写清楚。

## 6. 关键设计决策与理由

### 6.1 为什么主算法选择 LoRepair-SD 而不是图一致性修复？

LoState 可以包含 relation-annotated edges，也可以按需投影为轻量关系视图，但本文不将其定义为 LoStateGraph，也不把主方法建立在 GNN、graph diffusion 或 message passing 上。原因有三点：第一，本文贡献的核心是状态对齐与闭环验证，而不是图学习；第二，室内布局修复的第一版工程需求是可解释、可复验、可局部执行，LoRepair-SD 的 slot-instance matching、violation ranking、candidate region 和 acceptance controller 已经能覆盖主要违规类型；第三，图一致性传播需要额外 edit-subgraph 标注和训练成本，容易把论文主线从 reflective state interface 拉向 graph repair model。因此，图一致性修复可以作为 future extension，而 LoRepair-SD 是本文主算法。

### 6.2 为什么必须区分 Qwen-Image 与 Qwen2.5-VL？

Qwen-Image 和 Qwen2.5-VL 不是同一个模型。本文中二者的角色必须严格区分：

| 模型 | 本文角色 | 输入 | 输出 | 是否负责生成图像 |
|---|---|---|---|---:|
| Qwen-Image / Qwen-Image-ControlNet-Inpainting | 图像生成、编辑、局部 inpainting | 语义布局图、control mask、prompt | 新的 RGB 语义布局图 | 是 |
| Qwen2.5-VL | 状态构造、软约束验证、修正规划 | 图像 + LoState JSON + task prompt | JSON / 文本推理 / bbox_hint | 否 |

Qwen-Image 的技术报告将其定位为 image generation foundation model，并强调 complex text rendering 与 precise image editing；Qwen2.5-VL 的技术报告则强调视觉理解、object localization、structured data extraction 和 JSON/absolute-coordinate grounding。两者可以协同，但不能混写成一个模型（Qwen-Image Technical Report, 2025；Qwen2.5-VL Technical Report, 2025）。

### 6.3 为什么主设计采用同一 Qwen2.5-VL 做三角色？

主设计采用同一个 Qwen2.5-VL 权重承担：

1. **Target LoState Constructor**：从建筑 JSON、建筑平面图和用户需求构造 Goal LoState；
2. **Track B Reviewer**：基于 Goal–Observed LoState 判断软约束违规；
3. **Correction Planner**：把违规项转化为局部修正动作和期望结果 prompt。

这一设计可行，因为三者共享同一状态语言与同一空间推理基础：家具类别、bbox/mask、建筑元素、功能关系、约束列表和修正动作。它们不是三个独立视觉任务，而是同一 LoState 状态机中的三个推理位置。Qwen2.5-VL 已具备图像理解、layout/diagram analysis、object grounding 和结构化 JSON 输出能力；SpatialRGPT 等工作也说明区域级输入可以增强 VLM 的空间关系推理。因此，同一 VLM 通过 task prompt 切换角色在技术上可行，在论文表述上也更集中（Qwen2.5-VL Technical Report, 2025；SpatialRGPT, 2024）。

**实验上必须做消融**：

| 消融 | 目的 |
|---|---|
| Shared Qwen2.5-VL zero-shot prompt | 验证无需训练时是否可跑通 |
| Shared Qwen2.5-VL + multi-task LoRA | 主方法，验证共享权重是否有效 |
| Same backbone + task-specific LoRA | 检查共享 LoRA 是否存在负迁移 |
| Three independent VLMs | 检查独立专家是否带来性能上限 |
| LLM Target Constructor + VLM Reviewer/Planner | 对比“二角色 VLM”旧方案 |

论文中可以把“三个独立模型”作为性能对照，而不是主方法。若 shared VLM 性能略低但成本和统一性明显更好，可以强调 efficiency / unified reasoning；若 shared VLM 性能也更好，则创新性更强。

### 6.4 为什么 State Observer 的几何层必须程序化？

因为本文的输出是固定色板俯视语义布局图。对于这种图像，家具类别与边界可以通过 RGB → label map → connected components 确定性恢复。此时让 VLM 去重新识别 bbox 不仅多余，还会引入不必要的 grounding 偏差。正确分工是：

```text
程序负责：颜色映射、实例提取、bbox/mask、碰撞、越界、门窗遮挡、free floor、非目标漂移；
VLM 负责：目标构造、软约束解释、功能关系判断、修正规划。
```

这也是 LoState 能成为 verification-enabling state language 的核心原因：几何事实独立于生成模型和 VLM 推理，Reviewer 比对的是可复验状态，而不是模型自述。

### 6.5 为什么 VLM bbox 偏差不是核心问题？

如果直接让 Qwen2.5-VL 在图像上输出 bbox，它的 bbox 是视觉 grounding 结果，可能受缩放、边缘模糊、对象遮挡或坐标映射影响。Qwen2.5-VL 支持 absolute coordinate 和 JSON grounding，但这不意味着每个像素都可作为几何真值。本文的解决方式是：

1. 对已有对象：bbox/mask 从 label map 精确计算；
2. 对缺失对象：VLM 在 `mask_spec` 中给出语义目标区域，程序只依据 free-floor / obstacle / protected-instance 约束进行裁剪和合法性检查；
3. 对软约束证据：VLM 可输出 region hint，但最终 mask 仍由程序按 `mask_spec` 光栅化、二值化和保护非目标区域；
4. 对自然图像或非固定色板扩展：可另设开放式分割辅助，但不进入本文主路径。

因此，VLM bbox 的 ±10–20px 偏差不会成为主链路误差源；它只影响 optional region hint，而不影响已有家具实例的精确几何。

### 6.6 Qwen-Image-ControlNet-Inpainting 的 mask 是怎么给的？

Qwen-Image-ControlNet-Inpainting 接收的 mask 本质上是与输入图像对齐的单通道数据。本文把 mask 生成拆成两层：Correction Planner 决定 `mask_spec` 的语义意图，Mask Tensor Adapter 只执行几何栅格化与合法性检查。程序可以从 `mask_spec` 中的数值坐标、polygon、object mask 或 old-new union 直接生成 H×W mask：

```text
mask_spec.bbox              → rasterized rectangle mask
mask_spec.polygon           → rasterized polygon mask
mask_spec.instance_ref      → observed object mask
mask_spec.old_new_union     → union(old_instance_mask, proposed_region)
protected_refs              → subtraction from repaint region
architecture constraints    → clipping by boundary / clearance / free-floor
```

因此，“给数值构造 mask”是完全正确的工程做法，但语义上“修哪里”由 VLM Planner 的 `mask_spec` 决定；程序层只做确定性转换、边界裁剪、非目标保护和面积/尺寸检查。通常白色/1 表示重绘区域，黑色/0 表示保留区域。

### 6.7 为什么 ControlNet 条件仍然只有 control image、control mask 和 prompt？

这是当前 Qwen-Image-ControlNet-Inpainting pipeline 的实际接口约束。本文中的 `bbox_hint`、`target_zone`、`preservation_mask` 不作为模型原生输入，而是在进入模型前由 Mask & Prompt Constructor 转换为：

1. `control_mask`：实际重绘区域；
2. `control_image`：mask 后的布局图；
3. `prompt`：描述期望结果的文本。

非目标保持依赖三层机制：mask 几何保护、latent replacement 或后处理 compositing、re-parse 后的 non-target regression check。这样可以在现有 Qwen-Image-ControlNet 接口下实现可验证局部修正。

### 6.8 为什么 prompt 必须描述“期望结果”而非“操作过程”？

Inpainting 模型在 mask 区域内看不到可靠的原始内容，因此 prompt 不应依赖“从哪里移动到哪里”这种过程描述。更稳妥的写法是描述最终状态：

| ✅ 推荐 | ❌ 不推荐 | 原因 |
|---|---|---|
| Place a desk near the window. | Move the desk from the left to near the window. | mask 内旧位置不可见 |
| Place a bed against a wall away from the door. | Move the bed away from the door. | 正向目标更明确且不依赖方向编造 |
| Clear the doorway area. | Remove the desk and fix the old region. | 目标状态更适合 inpainting |
| Keep all other furniture unchanged. | Do not change anything else. | 与 protected_refs 和 preservation check 对齐 |

该设计来自 inpainting 信息流本身，也与 ControlNet 的空间条件控制思想一致（ControlNet, 2023；Diffusers Inpainting API, 2024）。

### 7.1 旧版本残留清理记录

| 旧残留 | 问题 | 本版处理 |
|---|---|---|
| `mask_plan` 与 `LoAction` 混用 | 与 LoState v1.4.9 中 RepairPlan + `mask_spec` 不一致 | 全文统一为 RepairPlan + `mask_spec`；`control_mask` 只作为执行张量 |

### 7.2 仍需你进一步确认的点

1. **Baseline 可跑性**：DiffuScene-N、SceneEditor-N、ReSpace、DirectLayout 是否都能获得代码/模型并转成统一 evaluation representation；若不能，主表需降级为“可复现 baseline + appendix discussion”。
2. **Qwen-Image-ControlNet-Inpainting 实现接口**：需要最终确认实际训练/推理 API 是否严格支持 `control_image + binary control_mask + prompt`，以及 fixed-palette 输出是否稳定。
3. **APM 资产恢复策略**：Final3DState 中 retrieved_model_id、height、z_offset、anchor 的恢复规则需要单独工程文档，否则 2D semantic layout 到 3D scene 会被审稿人质疑。
4. **Practical Usability 阈值校准**：必须用 training split 统计定阈值，不能凭经验手写阈值。
5. **Human study 问卷截图与协议**：需要提前准备统一渲染、随机左右、问题定义、样本数、bootstrap CI。
